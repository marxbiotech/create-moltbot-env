# Migration: 0.3.0 → 0.4.0

This migration adds multi-channel webhook bypass support to `sync-access.sh` (Slack + Telegram) and updates `ensure-queue.sh` default naming from Telegram-specific to generic webhook pattern.

## Prerequisites

- **moltbot-app with Slack webhook support** — If you use Slack, your app must handle the `/slack/events` route and have `SLACK_SIGNING_SECRET` configured as a wrangler secret. If you only use Telegram, this migration is still safe — sync-access.sh will skip Slack bypass creation when `SLACK_SIGNING_SECRET` is absent.

## Steps

### 1. Replace `scripts/sync-access.sh`

Replace `scripts/sync-access.sh` entirely with the latest version from the scaffold templates, then ensure it's executable:

```bash
chmod +x scripts/sync-access.sh
```

Key changes from the previous version:

- **Data-driven bypass configs** — Replaces hardcoded single `telegram-webhook` bypass with a `BYPASS_CONFIGS` array. Each entry is `name_suffix|path|secret_name`. Currently supports:
  - `telegram-webhook|/telegram/webhook|TELEGRAM_WEBHOOK_SECRET`
  - `slack-events|/slack/events|SLACK_SIGNING_SECRET`
- **Single secret list fetch** — Wrangler secrets are fetched once and shared across all bypass checks (previously there was only one bypass to check; the shared fetch avoids N calls for N bypass types)
- **Loop-based reconciliation** — Each bypass config is independently checked: creates if needed + secret exists, deletes if exists + secret removed, no-ops otherwise
- **Backward compatible** — Existing `{worker}-telegram-webhook` Access apps are recognized and preserved. No manual cleanup needed.

### 2. Update `scripts/ensure-queue.sh`

Three changes in the default auto-inject pattern (only affects environments without explicit `queues` config in `wrangler.jsonc`):

1. Default queue names: `{worker}-telegram` → `{worker}-webhook`, `{worker}-telegram-dlq` → `{worker}-webhook-dlq`
2. Default binding: `TELEGRAM_QUEUE` → `WEBHOOK_QUEUE`
3. Design Decision comment: `{worker}-telegram` → `{worker}-webhook`

If your overlays already have explicit `queues` config in `wrangler.jsonc` (which they should after running 0.3.0's ensure-queue), these defaults are never used and the change is a no-op.

If you have environments that rely on the auto-inject default, update their `wrangler.jsonc` queue config to use the new naming before upgrading:
- Rename queue binding from `TELEGRAM_QUEUE` to `WEBHOOK_QUEUE`
- Rename queue from `{worker}-telegram` to `{worker}-webhook`
- Rename DLQ from `{worker}-telegram-dlq` to `{worker}-webhook-dlq`

### 3. Replace `scripts/delete-env.sh`

Replace `scripts/delete-env.sh` entirely with the latest version from the scaffold templates, then ensure it's executable:

```bash
chmod +x scripts/delete-env.sh
```

Key change: bypass app cleanup now loops over the same `BYPASS_CONFIGS` array as `sync-access.sh`, deleting both `{worker}-telegram-webhook` and `{worker}-slack-events` Access apps (if they exist). Previously only deleted the single `{worker}-telegram-webhook` app.

### 4. Replace `.claude/commands/ensure-queue.md`

Replace `.claude/commands/ensure-queue.md` with the latest version from the scaffold templates.

Key changes:
- Queue creation description updated to reflect create-and-check-error approach (previously claimed script uses `wrangler queues list`)
- Removed `CLOUDFLARE_API_TOKEN` manual override suggestion (relied on undocumented behavioral inconsistency)

### 5. Update `CLAUDE.md`

In the Key Scripts section, update the `sync-access.sh` description:

```markdown
- `scripts/sync-access.sh` — reconciles Cloudflare Access Applications: checks current state, creates/deletes webhook bypass apps as needed (telegram, slack)
```

### 6. Update `.moltbot-env.json` version

```bash
tmp=$(mktemp) && jq '.meta.version = "0.4.0" | .meta.lastUpgrade = "'"$(date +%Y-%m-%d)"'"' .moltbot-env.json > "$tmp" && mv "$tmp" .moltbot-env.json
```

### 7. Verify

1. Run `make -C overlays/<any-env> sync-access` to verify multi-channel bypass detection works
2. Confirm existing `{worker}-telegram-webhook` Access app is recognized as "Already exists"
3. If Slack is configured, confirm `{worker}-slack-events` bypass app is created or detected
4. Verify `scripts/delete-env.sh` contains `BYPASS_CONFIGS` array (not hardcoded `telegram-webhook`)
